## План техтоку (≈15 хв) на основі демо `ollama_demo.py`

### 1. Вступ: що ми будуємо? (1–1.5 хв)

- **Контекст**: коротко, що таке твій «AI‑помічник.
- **Мета демо**: показати, як поєднати:
  - RAG (пошук по знаннях),
  - інструменти (`Tools` + Home Assistant),
  - LLM для аналітики ризиків (історія + свіжі дані).

### 2. Архітектура демо-проєкту (1.5–2 хв)

- **`MyBot` як фасад**:
  - ініціалізація ембеддера `SentenceTransformer("../MiniLM")`;
  - завантаження конфігів (`config.yaml`) і підключення до Home Assistant (`HassClient`).
- **`AirIntelligence` / `OpenAIAirIntelligence`**:
  - інтерфейс до різних моделей (локальна й хмарна),
  - однаковий API `process_request(...)`.
- **`Tools`**:
  - інкапсуляція доступу до зовнішніх систем (HASS, веб‑пошук, інше),
  - чому це краще, ніж захардкожені `if/else` у основному коді.

### 3. Організація RAG у `MyBot` (3 хв)

- **RAG-чанки для інтентів**:
  - список `intent_data`: «погода», «курс валют», «новини», «скільки коштує».
  - для кожного — ембеддинг + метадані: `tool`, `query`, `params`, `embedding`.
- **RAG по кімнатах / сенсорах**:
  - запит до Home Assistant: `areas()` → назви кімнат.
  - генерація тексту типу: «Я хочу дізнатися температуру у приміщенні спальня…».
  - ембеддинг кожного такого тексту → семантичний індекс під сенсори.
- **Пошук кращого тулу**:
  - `_get_best_tool(user_text)`: `encode` → `cosine_similarity` → максимальна схожість.
  - якщо поріг `threshold` пройдено — знаходимо відповідний тул/запит/параметри.
  - приклад: юзер каже «яка температура у спальні» → знайдеться RAG‑запис з температурою + кімнатою.

### 4. Механіка виклику тулів (2 хв)

- **Пайплайн `ask(...)`**:
  - будуємо `local_kb` (локальну базу знань для конкретної відповіді).
  - якщо `user_context` передано явно (наприклад, `{"messages": ...}`) — використовуємо його.
  - інакше — запускаємо `_get_best_tool(...)`, викликаємо `bot.tools.tool_<tool_id>`.
- **`Tools` як шар абстракції**:
  - чітке розділення: LLM думає про «що/чому», тулзи — про «як/де взяти дані».
  - легке додавання нових тулів без змін в основній логіці.

### 5. Few-shot vs Zero-shot у цьому демо (2–2.5 хв)

- **Zero-shot**:
  - LLM отримує системний промпт + контекст (математика + повідомлення) без прикладів.
  - **Плюси**: простіше, гнучкіше, менше коду, легше змінювати завдання.
  - **Мінуси**: іноді менш стабільні формати відповідей, доводиться описувати формат словами.
- **Few-shot (де могли б використати)**:
  - для стабільної оцінки ризику (`Low/Medium/High/Critical`) можна додати 2–3 приклади:
    - як виглядає відповідь при `Low`,
    - як при `High/Critical` із короткими коментарями.
  - місце інтеграції: у `military_prompt` додати блок «Приклади аналізу…».
- **Компроміс у демо**:
  - показати, що навіть zero-shot + хороший контекст дають вже корисну аналітику,
  - а few‑shot — наступний крок, коли хочемо продакшн‑стабільність.

### 6. RAG + моніторинг + математика: сценарій демо (3–3.5 хв)

- **Крок 1 — історичні дані (`wiki_to_csv` + `calculate_next_strike`)**:
  - парсимо сторінку Wikipedia з ударами;
  - зберігаємо у CSV;
  - рахуємо «математичний» прогноз наступного удару;
  - виводимо звіт (`math_report`) в консолі.
- **Крок 2 — свіжі сигнали (`scrape_messages`)**:
  - скрейп публічного Telegram‑каналу;
  - важливість: додає «real‑time» сигнали, які не видно з історії.
- **Крок 3 — LLM як військовий аналітик**:
  - системний промпт: «Ти — військовий аналітик…».
  - передаємо в `user_context`:
    - `math_report` (історичний/математичний шар),
    - `messages` (свіжі повідомлення).
  - просимо оцінити ризик, знайти ознаки підготовки, дати рівень загрози.
- **Підсумок**: LLM не замінює ППО, але допомагає:
  - об’єднати різні джерела;
  - дати людинозрозумілу оцінку ризику.

### 7. Q&A + ідеї для розвитку (1–1.5 хв)

- **Можливі напрямки розвитку**:
  - додати few-shot приклади для більш стабільних відповідей;
  - розширити RAG базу (інші джерела, багатомовність);
  - інтегрувати алерти в Home Assistant / мобільний нотифай;
  - додати простий UI замість CLI.
- **Питання від аудиторії**:
  - про моделі (`MiniLM`, OpenAI/локальні);
  - про безпеку та якість джерел;
  - про продакшн‑обмеження (rate-limits, конфіденційність тощо).
